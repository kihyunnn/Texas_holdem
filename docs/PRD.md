# 텍사스 홀덤 게임 트래커 - 제품 요구사항 명세서 (PRD)

## 1. 프로젝트 개요

### 1.1 목적
친구들끼리 텍사스 홀덤 게임을 즐길 때 사용할 수 있는 간단한 게임 기록 및 통계 웹 애플리케이션

### 1.2 목표 사용자
- 친구들과 정기적으로 텍사스 홀덤 게임을 하는 사람들
- 게임 기록과 승률을 추적하고 싶은 플레이어들
- 홀덤 족보를 자주 참고해야 하는 초보 플레이어들

### 1.3 핵심 가치 제안
- **간편한 기록**: 매 판마다 누가 이겼는지, 얼마를 땄는지 빠르게 기록
- **투명한 통계**: 플레이어별 승률, 수익/손실을 실시간으로 확인
- **편리한 참고**: 게임 중 언제든 족보를 확인 가능
- **쉬운 배포**: Docker로 홈서버에 간단하게 배포 가능

## 2. 기능 요구사항

### 2.1 필수 기능 (MVP)

#### 2.1.1 플레이어 관리
- [ ] 플레이어 추가 (이름만 입력)
- [ ] 플레이어 목록 조회
- [ ] 플레이어별 통계 조회

#### 2.1.2 게임 기록
- [ ] 게임 결과 입력
  - 참여 플레이어 선택 (다중 선택)
  - 각 플레이어의 베팅 금액 입력
  - 승자 선택
  - 팟 금액 (총 판돈) 입력
  - 메모 (선택사항)
- [ ] 게임 기록 저장 (SQLite)
- [ ] 최근 게임 기록 조회

#### 2.1.3 통계 및 승률 확인
- [ ] 리더보드
  - 순이익 순으로 정렬
  - 플레이어명, 참여 게임 수, 승리 횟수, 순이익, 승률 표시
- [ ] 개별 플레이어 통계
  - 총 참여 게임 수
  - 총 승리 횟수
  - 총 베팅 금액
  - 총 획득 금액
  - 순이익/손실
  - 승률 (%)

#### 2.1.4 홀덤 족보 참고
- [ ] 텍사스 홀덤 핸드 랭킹 테이블
  - 로열 플러시부터 하이카드까지 순서대로 표시
  - 각 족보의 설명
  - 예시 카드 조합

### 2.2 제외 기능 (Out of Scope)
- ❌ AI 기능
- ❌ 사용자 인증/로그인
- ❌ 멀티플레이어 실시간 게임
- ❌ 베팅 시뮬레이션
- ❌ 복잡한 통계 분석 (그래프, 트렌드 분석 등)

## 3. 비기능 요구사항

### 3.1 성능
- 페이지 로딩 시간: 2초 이내
- API 응답 시간: 500ms 이내
- 동시 사용자: 10명 이하 (친구들끼리 사용)

### 3.2 사용성
- 모바일 친화적 반응형 디자인
- 직관적인 UI (설명 없이 사용 가능)
- 한글 인터페이스

### 3.3 보안
- SQL Injection 방지 (Parameterized Query 사용)
- CORS 설정 (필요시)

### 3.4 확장성
- 100개 이상의 게임 기록 처리 가능
- 10명 이상의 플레이어 관리 가능

### 3.5 배포성
- Docker 컨테이너화
- docker-compose로 한 번에 실행 가능
- 홈서버 배포 지원

## 4. 기술 스택

### 4.1 백엔드
- **프레임워크**: Flask (Python)
  - 이유: 가볍고 간단, 빠른 개발 가능
- **데이터베이스**: SQLite
  - 이유: 파일 기반, 별도 DB 서버 불필요, 설정 간편
- **프로덕션 서버**: Gunicorn
  - 이유: 안정적인 WSGI 서버

### 4.2 프론트엔드
- **HTML5**: 구조
- **CSS3**: 스타일링 (Vanilla CSS)
- **JavaScript (ES6+)**: 동적 기능
- **라이브러리**: 없음 (빌드 프로세스 제거)

### 4.3 배포
- **컨테이너**: Docker
- **오케스트레이션**: Docker Compose
- **OS**: Linux (홈서버)

## 5. 데이터 모델

### 5.1 Players (플레이어)
```
- id: INTEGER PRIMARY KEY
- name: TEXT (UNIQUE)
- created_at: TIMESTAMP
```

### 5.2 Games (게임)
```
- id: INTEGER PRIMARY KEY
- winner_id: INTEGER (FK -> players.id)
- pot_amount: INTEGER (총 판돈)
- played_at: TIMESTAMP
- notes: TEXT (메모)
```

### 5.3 Game_Participants (게임 참여자)
```
- id: INTEGER PRIMARY KEY
- game_id: INTEGER (FK -> games.id)
- player_id: INTEGER (FK -> players.id)
- bet_amount: INTEGER (베팅 금액)
```

## 6. 사용자 시나리오

### 6.1 첫 사용
1. 홈서버에 Docker Compose로 애플리케이션 배포
2. 브라우저로 접속 (http://서버IP:5000)
3. 플레이어 추가 (친구들 이름 등록)
4. 게임 시작

### 6.2 게임 진행 중
1. 한 판이 끝날 때마다 "게임 기록" 탭 선택
2. 참여한 플레이어들 체크
3. 각자 베팅한 금액 입력
4. 총 판돈 입력
5. 승자 선택
6. "기록" 버튼 클릭
7. 족보가 햇갈리면 "족보" 탭에서 확인

### 6.3 게임 종료 후
1. "통계" 탭 선택
2. 리더보드에서 오늘의 승자 확인
3. 각 플레이어의 수익/손실 확인

## 7. UI/UX 요구사항

### 7.1 디자인 원칙
- **단순함**: 복잡한 기능 없이 핵심 기능만
- **빠름**: 클릭 최소화, 빠른 입력
- **명확함**: 버튼과 입력란의 용도가 명확
- **모던함**: 다크 모드, 그라디언트, 부드러운 애니메이션

### 7.2 레이아웃
- 탭 기반 네비게이션 (게임 기록 / 통계 / 족보)
- 모바일에서도 사용 가능한 반응형 디자인
- 카드 기반 UI

### 7.3 컬러 스킴
- 다크 모드 기본
- 포커 테마 (그린, 레드, 골드 악센트)
- 고대비 (가독성 우선)
